# Refactoring Plan: Cash & Financial Transactions

## 1. Current State (Gaps)

- Transactions in `CashService` and `useShift` occasionally store redundant string data (like `employeeName` or `saleDescription`) instead of relying on unique identifiers.
- Reports and cash drawer reconciliation logic might fragment if a user's name or a sale's metadata changes, as the "link" is sometimes a static string.

## 2. Infrastructure Changes

- **Primary Key:** Every financial entry MUST use the `TRX-XXXX` ID generated by `idGenerator.generate('transactions')`.
- **Foreign Keys:**
  - `shiftId`: Strict reference to the GUID of the open shift.
  - `userId`: Strict reference to the `employeeId` (e.g., `EMP-0001`).
  - `relatedEntityId`: Instead of a generic description, use this field to store `saleId` or `purchaseId`.

## 3. Implementation Steps

1. **Refactor Transaction Interface:** Update `Transaction` type to strictly require `userId` and `shiftId`.
2. **Simplified Handlers:** Update `addTransaction` in `useShift.ts` to accept only the `amount`, `type`, and `relatedId`. The hook should automatically inject the `currentShift.id` and `currentEmployeeId`.
3. **Dynamic View Layer:** In the Finance Dashboard, use the `userId` to lookup the current name from the `employees` data slice instead of reading a static name from the transaction object.

## 4. Expected Outcome

- 100% data consistency in financial auditing.
- Ability to track a specific employee's cash handling across multiple shifts even if their display name is updated.
- Cleaner, sharded storage for financial records.
