# خطة تطوير وتحديث ملف التثبيت المتقدم لبرنامج ZINC (Tauri & NSIS Advanced Installer Roadmap)

هذا المستند يقدم الخطة المعمارية المتقدمة والتنفيذية لتطوير ملف التثبيت (NSIS Installer) ونظام توزيع التطبيق لبرنامج **ZINC (PharmaFlow AI)** لملاءمة متطلبات العمل الفعلي داخل الصيدليات والشركات.

---

## 🟢 1. الميزات التي تم تنفيذها بنجاح (Completed)

### 1.1 النسخ الاحتياطي التلقائي عند التحديث (Pre-Install Snapshot)
* **الوصف:** يقوم ملف التثبيت بفحص العمليات النشطة، إغلاق البرنامج بأمان، وأخذ نسخة احتياطية من البيانات المحلية قبل التحديث.
* **الحالة:** تم التنفيذ (NSIS Hooks).

### 1.2 التهيئة التلقائية للملحقات والجدار الناري (Firewall Auto-Provisioning)
* **الوصف:** فتح منافذ الطباعة الحرارية (Port 9100) وتفعيل خدمة Print Spooler تلقائياً دون تدخل يدوي.
* **الحالة:** تم التنفيذ (NSIS Hooks).

### 1.3 نظام التحديثات الصامتة (Silent Background Updater)
* **الوصف:** تنزيل التحديثات في الخلفية باستخدام `tauri-plugin-updater` وتطبيقها بسلاسة عبر واجهة المستخدم أو عند إعادة التشغيل.
* **الحالة:** تم التنفيذ (React Hooks & Rust Backend).

---

## 🚀 2. ميزات متقدمة مقترحة لملف التثبيت نفسه (NSIS Installer Advanced Features)

هذه الميزات تتعلق بملف الـ `Setup.exe` نفسه لتحويله من مجرد ناسخ ملفات إلى مساعد ذكي لتهيئة الصيدلية:

### 2.1 الفحص الاستباقي لبيئة الكاشير (Environment Health Check)
* **الهدف:** منع تثبيت البرنامج على أجهزة ضعيفة جداً أو أنظمة تشغيل غير مدعومة لتجنب الشكاوى التقنية لاحقاً.
* **آلية التنفيذ:**
  * يقوم NSIS قبل بدء التثبيت بفحص: مساحة القرص C، حجم الرامات (أقل شيء 4GB)، وإصدار الويندوز (تحذير إذا كان Windows 7).
  * فحص سرعة واستقرار الإنترنت (Ping Test لـ Supabase) لتنبيه الصيدلي إذا كانت شبكته غير مستقرة.

### 2.2 الهجرة التلقائية من الأنظمة القديمة (Legacy Systems Auto-Migration)
* **الهدف:** تسهيل انتقال الصيدليات من البرامج القديمة (مثل المتمدد، فارما فلاي، الصيدلي) إلى ZINC بضغطة زر.
* **آلية التنفيذ:**
  * يكتشف NSIS وجود مجلدات وقواعد بيانات البرامج القديمة المعروفة في مساراتها الافتراضية.
  * يعرض شاشة (Custom NSIS Page) تسأل المستخدم: *"تم اكتشاف نظام قديم، هل ترغب في سحب الأرصدة والعملاء تلقائياً؟"*.
  * يمرر ملف التثبيت مسار البيانات القديمة كمتغير (Command Line Argument) لتطبيق ZINC ليعالجه عند أول تشغيل.

### 2.3 روابط التثبيت السحرية للفروع (Magic Link / Multi-Tenant Install)
* **الهدف:** إراحة مدير الصيدليات ذات الفروع المتعددة من تسجيل الدخول يدوياً في كل فرع.
* **آلية التنفيذ:**
  * يتم تحميل ملف التثبيت باسم مخصص، مثل `zinc_setup_branch102.exe` أو تمرير Token.
  * يقرأ NSIS اسم الملف أو الـ Token ويزرعه في ملف `app-config.json` في مجلد AppData، بحيث يفتح البرنامج لأول مرة وهو مسجل دخول كفرع "102" متصل بالفرع الرئيسي تلقائياً.

### 2.4 اكتشاف طابعات الشبكة تلقائياً (Network Hardware Auto-Discovery)
* **الهدف:** عدم مطالبة المستخدم بضبط إعدادات الـ IP الخاصة بطابعة الفواتير.
* **آلية التنفيذ:**
  * أثناء شريط التحميل (Progress Bar)، يقوم سكريبت فرعي بعمل فحص (ARP Scan أو UDP Broadcast) لاكتشاف طابعات الإيصالات الحرارية المتصلة بالراوتر.
  * يقوم بإنشاء منفذ طابعة وهمي (Standard TCP/IP Port) في الويندوز وتعريف الطابعة تلقائياً عبر سكريبت PowerShell مخفي.

### 2.5 وضع الإغلاق الإجباري للكاشير "Kiosk Mode Setup"
* **الهدف:** تحويل جهاز الكمبيوتر العادي إلى جهاز POS مقفول على تطبيق ZINC فقط.
* **آلية التنفيذ:**
  * إضافة خيار في NSIS: "تخصيص هذا الجهاز ككاشير مغلق (Kiosk)".
  * يقوم NSIS بتعديل الـ Registry لإضافة ZINC كبديل لـ `explorer.exe` (Windows Shell) للمستخدم الحالي، بحيث يفتح الكمبيوتر مباشرة على ZINC ولا يمكن للكاشير فتح المتصفح أو الألعاب.

## 💡 3. ميزات متقدمة مقترحة لتطبيق الكاشير (Tauri App Advanced Features)

### 3.1 محرك المبيعات الأوفلاين (Offline-First Failover via SQLite)
* **الهدف:** عدم توقف المبيعات إطلاقاً عند انقطاع الإنترنت عن الصيدلية.
* **آلية التنفيذ:** استخدام `tauri-plugin-sql` محلياً. تحفظ الفواتير وقت الانقطاع، وترفع لـ Supabase فور عودة الاتصال.

### 3.2 دعم الشاشة المزدوجة للعميل (Customer Facing Display)
* **الهدف:** عرض إجمالي الفاتورة وعروض ترويجية للعميل أثناء وقوفه أمام الكاشير.
* **آلية التنفيذ:** اكتشاف الشاشة الثانية برمجياً وفتح نافذة إضافية (Frameless) تلقائياً لمزامنة سلة المشتريات.

### 3.3 التكامل المباشر مع موازين التركيبات (Serial Peripherals)
* **الهدف:** قراءة أوزان التركيبات الطبية تلقائياً.
* **آلية التنفيذ:** استخدام `tauri-plugin-serialport` للاتصال المباشر بمنافذ الـ COM الخاصة بالميزان الإلكتروني.

### 3.4 نظام التقاط الأخطاء والقياس عن بعد (Telemetry & Crash Reporting)
* **الهدف:** تتبع أعطال البرنامج وحلها دون إزعاج الصيدلي لطلب الـ Logs.
* **آلية التنفيذ:** دمج Sentry في بيئة Rust و React لرفع أي عطل (Panic) مع مواصفات الجهاز لخوادم التطوير مباشرة.
