# ZINC Clean Code & Architecture Guidelines 🚀

هذا المستند يلخص الأسلوب المتبع في تطوير مشروع **ZINC OS** لضمان كتابة كود احترافي، نظيف، وعالي الأداء.

---

## 1. فلسفة "تسطيح الكود" (Flatten the JSX Tree) 🌳
تجنب تداخل الـ Divs (Div Soup). كلما قل عمق الـ DOM، زاد الأداء وسهلت القراءة.
*   **القاعدة**: استخدم `Gap` و `Padding` و `Layout Utilities` في العنصر الأب بدلاً من إنشاء حاويات جديدة.
*   **التطبيق**: ادمج العناصر المتقاربة وظيفياً في حاوية واحدة.

## 2. المنطق المحسوب (Smart Memoization) 🧠
لا تضع عمليات حسابية معقدة داخل الـ JSX.
*   **القاعدة**: استخدم `useMemo` لتجهيز البيانات قبل عرضها.
*   **الهدف**: إبقاء الـ JSX "نظيفاً" ومخصصاً للعرض فقط، مع ضمان عدم إعادة العمليات الحسابية إلا عند تغير البيانات الفعلية.

## 3. الاختصار الذكي (Concise JSX Patterns) ✂️
استخدم اختصارات لغة JavaScript و React لتقليل عدد الأسطر.
*   **القاعدة**: استخدم `Implicit Returns` (مثل `() => (...)`) للمكونات البسيطة.
*   **القاعدة**: استخدم `Conditional Rendering` الذكي (مثل `src={darkMode ? 'white.svg' : 'black.svg'}`) بدلاً من تكرار العناصر.

## 4. برمجة المجموعات (Functional Iteration) 🔄
تجنب تكرار العناصر المتشابهة يدوياً.
*   **القاعدة**: استخدم المصفوفات والـ `map` لتوليد العناصر المتكررة (مثل الـ Particles أو القوائم).
*   **الفائدة**: سهولة تعديل كل العناصر من مكان واحد وتقليل احتمالية الخطأ.

## 5. تجريد المكونات (Component Extraction) 📦
ملف `App.tsx` هو المايسترو، لا يجب أن يحتوي على تفاصيل التنفيذ المعقدة.
*   **القاعدة**: أي جزء من الواجهة له منطق مستقل (مثل الـ Logout Overlay أو Landing Page) يجب أن يُفصل في ملف خاص.
*   **الهدف**: تقليل حجم الملفات الأساسية وتسهيل عملية الـ Debugging.

## 6. التنسيق بالسمات (Theme-Driven Styling) 🎨
الاعتماد على نظام Tailwind و CSS Variables.
*   **القاعدة**: لا تستخدم ألواناً صريحة (Hardcoded) إلا للضرورة القصوى. اعتمد على `zinc-500`, `primary-500` والـ `dark:` variants.
*   **النتيجة**: نظام مرن يدعم الـ Dark Mode وتغيير السيمات بسهولة.
